## Why this sprint exists

The Figma design (9 screens, see `docs/design/figma/`) was finalised after Sprint B4 closed. Walking through the screens against the current BE surface revealed a small set of gaps that, if left until the FE sprint, would force mid-sprint backend reworks and break the BE-first contract. B4.6 closes those gaps so the customer FE template can wire data 1:1 with no surprises.

This is **schema + service tweaks**, not new features. Scope is deliberately small.

## Why pause B5

Sprint B5 (hardening + Railway deploy) was originally the next stop. Deploying the BE alone is technically correct but operationally costly: every time the FE discovers a missing field, an off-by-one query, or a wrong CORS host, we redeploy. Better to:

1. Finish B4.6 (this sprint) so the API surface matches the design.
2. Build customer FE locally against `localhost:3000`.
3. Build admin FE locally against `localhost:3000`.
4. **Then** Sprint B5 — deploy all three repos together as one validated system.

Local-first works because Supabase/Stripe/Resend are already remote-hosted; the only thing on `localhost` is the Node process and the two Next.js dev servers.

## Sub-features

### B4.6.1 — Add `Tour.isFeatured`

```prisma
model Tour {
  // ...existing fields
  isFeatured  Boolean  @default(false) @map("is_featured")

  @@index([isFeatured, isPublished])  // home-page hero strip query
}
```

Drives:

- Home page "Trending Tour Packages" strip (3-card row in the Figma home screen).
- `?featured=true` filter on `GET /tours` (B4.6.3).

> **Shipped:** combined into the single migration `20260514110347_b4_6_figma_alignment` (replaces the old single-column `tours_is_featured_idx` with the composite `(is_featured, is_published)` index).

### B4.6.2 — Extend `TourCategory` enum

```prisma
enum TourCategory {
  DAY
  PACKAGE
  CUSTOM
  HONEYMOON   // new
  MUSICAL     // new
}
```

Drives the Services dropdown in the Figma nav (Honeymoon Packages / Tours Packages / Musical Events / Build Package). `Build Package` is a separate flow — see BACKLOG.

Postgres enum extension is non-destructive (`ALTER TYPE ... ADD VALUE`), so this is safe on existing data.

> **Shipped:** both `ADD VALUE` statements live in the same combined migration `20260514110347_b4_6_figma_alignment` as B4.6.1.

### B4.6.3 — Query DTO: `featured` + sort

**Original plan** — a single packed `sort` enum:

```ts
sort?: 'createdDesc' | 'priceAsc' | 'priceDesc' | 'titleAsc' | 'titleDesc';
```

> **Shipped (diverged):** the DTO already exposed a cleaner **2-axis** pair, so B4.6.3 reused it instead of adding the packed enum:
>
> ```ts
> class ListToursQueryDto {
>   // existing: destination, category, minPrice, maxPrice, duration, page, limit
>   featured?: boolean;
>   sortBy?: 'createdAt' | 'basePrice' | 'durationDays' | 'titleEn' = 'createdAt';
>   sortOrder?: 'asc' | 'desc' = 'desc';
> }
> ```
>
> Still maps to the Figma sort tabs ("Date / Price Low To High / Price High To Low / Name A-Z") — the FE composes `sortBy` + `sortOrder` per tab. Default stays newest-first (`createdAt` / `desc`).

### B4.6.4 — Card payload enrichment

The Figma tour cards show:

- Average rating (★ 5.0)
- "120+ People" travelling

Current `GET /tours` returns the raw tour row only. Add to each card:

```ts
{
  // existing fields
  averageRating: number | null,   // null when no approved reviews
  reviewsCount: number,           // approved only
  peopleGoing: number,            // sum of seats across all PAID bookings on the tour's departures
}
```

Implementation: extend the existing `findPublishedList` query with aggregated subqueries or a `$queryRaw` join. `reviews(tourId, isApproved)` index already exists; for `peopleGoing` we sum `tour_departures.seats_booked` (already maintained by the Stripe webhook + admin refund flow).
